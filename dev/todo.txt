ROADMAP:
- Generational GC notes:
  - Addresses now actually change, so we can't easily use them as
    say hashtable keys.
  - Generational gc can still guarantee realtime performance,
    if there are enough generations. if each next older generation
    is 2 times slower to gc, then with 16 generations entire 32-bit address
    space should be allocable. just make sure really old generations are
    never GCed, if we have not enough time for that.
  - One major problem with the generational GC is some objects, like the open
    files, which can persist for enough cycles to be considered permanent and
    moved into the permanent generation. So instead of a permanent root
    generation, we need the explicitly permanent objects as such. We can no
    longer hold pointers to objects, since they can become invalid at any
    moment, not just when the function returns.
    As of now the problem is solved by pre-allocating such objects and reusing
    them.

- Convert Symta to use generational GC:
  - Implement dynamically resizeable heap, so we will take only
    as much memory as we need.
    That can be implemented either by changing the upper heap to grow downwards
    and remapping empty pages on the go, or by copying everything and doubling
    size.
  - GC can be optimized by checking for at the interpreter level
    i.e. after each Nth method call or jump.
  - printing should disable gc()
    so as_text will produce proper Me^address.x
  - Closure captures the entire argument lists, even if some args are unused.
    Not only does that waste memory, but also leads  to subtle and
    hard to debug memory leaks..
    Divise a better closure scheme.
  - theap must stay, because of TG_DIRTY flags.
    These cant be stored inside the heap itself, since we don't
    know what is object and the cost of parsing 64-bit values instead of 1-bit
    flags.
  - Runtime should have a set of builtin tests to ensure GC works as intended.
    I.e. for each generations size, do a number of allocs of different types.
    including views
  - CONS could turn into normal list on GC or on get/set.
  - we need a gc_raw array, which would allow holding any type of data.
  - by malloc'ing the api_t, we can attach a dynamic array, like pgmod
    to its end.
  - When we are out of memory, it is possible to allocate new,
    larger generation with malloc, but it will really complicate the theap0
    heap0 and pgmod use. Proper implementation would reserve a page range
    and then dymalically add pages to its end.
  - Page tables can also be used to replace size,
    if each generation gets a page sized allocation pools for smaller objects.
    That change is mandatory, if we want to host integers that way.
  - While walking the stack frames, we have to min/max the pointer location
    we have relocated. That way we wont have to walk entire stack for each gen.
    Then we will also have to save the max location since the last GC.
  - Both heaps growing towards center can be accomplished by redefining
    the api_t.alloc pointer during the generation switch. But it will create
    problems with the "INSIDE_GENERATION" check
  - Oldest generation may use malloc and free for larger objects,
    since objects are unlikely to be moved.
  - Finalizer pointer can reside inside gc_obj_head_t.code
    But something like the lifts list will be needed for finalizers.
    Or at least a page table.
  - Collecting garbage without a pause, would require providing a temporary
    generation for the running thread, which is being collected.
  - Growing heaps. That would allow for lightweight threads.
    And most references to heap are done through hgp anyway.

Transition to 32-bit pointers:
  - First replace uintptr_t with uint64_t or an equivalent type,
    which is guaranteed to be always 64-bit
  - We need tags, but can't have 5-byte values, since many architectures
    don't allow unaligned access, while keeping tags in a separate 1-byte
    array would incur two memory accesses, and require very tricky code
    to copy both the value and the tag on stack, while it will be kept as a
    64-bit value anyway. Even more problems result from the requirement to
    reimplement fixtext and doubles. 64-bits are simply mandatory.
    Therefore the best solution is to use 64-bit fixnums even on 32-bit systems.
  - FIXNUM are mandatory, since replacing them with memory accesses will incur
    heavy penalty, while `if` depends on them being open coded,
    with fixnum 0 being 0
  - Instead of void, use dyn everywhere. And dyn should be defined as
    uint64_t.
